# 🎯 Overview

## Business Context

VietDist is a simulated FMCG distribution company operating through multiple business functions, including sales, customer management, distributor operations, promotion campaigns, and sales target planning.

The company generates operational data from multiple source systems in different file formats and refresh frequencies. These datasets are not suitable for direct analytical reporting and require a structured data warehousing solution to ensure data consistency, quality, and scalability.

## Project Objective

The objective of this project is to design and implement an end-to-end Sales Data Warehouse for VietDist using Python, PostgreSQL, and SQL.

The solution aims to:

* Integrate operational data from multiple business domains into a centralized PostgreSQL database.
* Build an ETL pipeline following the Bronze–Silver–Gold architecture.
* Standardize and validate raw datasets through data profiling and quality checks.
* Implement historical sales target versioning and Slowly Changing Dimension (SCD Type 2).
* Design dimensional models and business-ready data marts to support sales performance and distributor analysis.

# 🥉 Bronze Layer 

## Objective

The Bronze layer is responsible for ingesting raw operational datasets into PostgreSQL while preserving the original data structure and ensuring full data traceability.

No business transformations are applied at this stage. All source records are loaded exactly as received to maintain data integrity and support future auditing.

---

## Source Data

The Bronze layer ingests data from 10 operational datasets provided by different business functions within VietDist.

Supported file formats include:

- CSV
- XLSX
- XLSM
- XLSB

---

## Metadata Tracking

Each ingestion batch records metadata to improve data lineage and traceability.

### Ingestion Metadata

| Column | Description |
| :----- | :---------- |
| `_source_file` | Original source file name |
| `_source_platform` | Source system identifier |
| `_ingested_at` | Timestamp when data was loaded |
| `_batch_id` | Batch identifier |

Additionally, ingestion activities are logged using dedicated metadata tables:

- `raw.ingest_log`
- `raw.sales_target_files`

These tables store information such as batch status, number of rows loaded, loading duration, and error messages.

---

## Special Case: Sales Target Versioning

Unlike other datasets, the **sales_target_plan** source contains multiple planning versions.

Instead of overwriting historical targets, every version is preserved in the Bronze layer for downstream version management.

| Planning Version | Description |
| :--------------- | :---------- |
| Plan_v1_Original | Initial annual sales target |
| Plan_v2_Adjustment_H2 | Mid-year adjustment for H2 |

This design enables historical comparison between original plans and revised targets in later transformation stages.

# 🥈 Silver Layer 

## Objective

The Silver layer transforms raw operational data into clean, standardized, and analytics-ready datasets.

This layer focuses on improving data quality through profiling, cleansing, validation, and business rule implementation before dimensional modeling in the Gold layer.

---

## Data Profiling

Before applying any transformations, all raw datasets were profiled to evaluate data quality and identify potential issues.

Profiling activities included:

- Row count validation
- NULL value analysis
- Duplicate detection
- Primary key validation
- Business rule validation

All profiling findings and planned remediation actions are documented in:

```text
docs/data_issues.md
```

---

## Data Cleaning & Standardization

After profiling, datasets were transformed into staging models with consistent structures and validated business rules.

Key transformations include:

- Standardized column names
- Data type casting
- NULL value handling
- Duplicate removal
- Date format standardization
- Primary key validation
- Business rule implementation

The goal of this stage is to ensure that downstream analytical models are built on trusted and consistent data.

---

## Sales Target Versioning

The **sales_target_plan** dataset contains multiple planning versions released throughout the year.

Instead of keeping only the latest target, the Silver layer preserves historical versions by implementing versioning logic.

Business rules:

| Period | Effective Version |
| :----- | :---------------- |
| January – June | Version 1 |
| July – December | Version 2 |

Additional attributes were generated to support historical analysis:

| Column | Description |
| :----- | :---------- |
| `effective_from` | Version effective start date |
| `effective_to` | Version effective end date |
| `is_latest` | Indicates the latest effective version |

This design enables historical target tracking while maintaining accurate comparisons between planned and actual sales performance.

---

# 🥇 Gold Layer 

## Objective

The Gold layer transforms standardized staging data into a dimensional data model optimized for analytical reporting.

Following **Kimball's Dimensional Modeling** approach, the Gold layer organizes business data into **dimension tables**, **fact tables**, and **business marts**, providing a scalable foundation for performance analysis and future BI dashboards.

---

## Dimension Tables

Dimension tables provide descriptive business attributes used for filtering, grouping, and reporting.

| Table | Description |
| :---- | :---------- |
| `dim_date` | Calendar dimension with fiscal year, month, quarter, and week attributes. |
| `dim_customers` | Customer master information for sales analysis. |
| `dim_products` | Product catalog, categories, and pricing information. |
| `dim_employees` | Sales representative dimension with **Slowly Changing Dimension (Type 2)** implementation. |
| `dim_distributors` | Distributor dimension supporting historical business changes using **SCD Type 2**. |

### Slowly Changing Dimension (Type 2)

Historical changes to employee and distributor information are preserved using **SCD Type 2**.

Instead of overwriting existing records, a new record is created whenever tracked attributes change.

Additional columns include:

- `effective_from`
- `effective_to`
- `is_current`

This enables historical reporting by matching fact records with the correct dimension version at the transaction date.

---

## Fact Tables

Fact tables store measurable business events and are linked to dimension tables through surrogate keys.

| Table | Business Grain |
| :---- | :------------- |
| `fact_sales` | One record per sales transaction |
| `fact_returns` | One record per return transaction |
| `fact_targets` | One employee × one month × one target version |

---

## Business Data Marts

The project delivers two analytical marts designed to support business reporting.

### mart_sales_vs_target

Compares actual sales performance against planned sales targets.

Key KPIs:

- Actual Revenue
- Target Revenue
- Revenue Variance
- Achievement Percentage

Business use cases:

- Monitor employee performance
- Compare planned vs actual sales
- Track monthly sales achievement

---

### mart_distributor_performance

Aggregates distributor performance metrics for operational analysis.

Key KPIs:

- Total Revenue
- Total Orders
- Return Amount
- Average Order Value

Business use cases:

- Evaluate distributor performance
- Compare regional distributors
- Identify high-performing distribution partners
