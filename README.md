# 📌 Background & Business Problem

## Business Context

VietDist is a nationwide FMCG distribution company operating through multiple sales channels, distributors, customers, and sales representatives.

Business stakeholders need a centralized analytical layer to answer key business questions:

* How much revenue is generated by each employee?
* Are employees achieving their assigned sales targets?
* Which distributors perform best?
* Which products generate the highest revenue?
* How do return transactions impact business performance?

The source data is distributed across multiple CSV and Excel files, making reporting difficult and inconsistent.

---

# 🎯 Project Objective

Build an end-to-end Data Warehouse using the Bronze–Silver–Gold architecture.

The solution aims to:

* Centralize operational data into PostgreSQL
* Standardize and clean raw datasets
* Support historical sales target versioning
* Implement dimensional modeling using Star Schema
* Deliver business-ready data marts for analytical reporting

# 🏗️ Solution Architecture

The platform follows a Medallion Architecture pattern:

```text
Raw Files (CSV / Excel)
          ↓
     Bronze Layer
   (Raw Ingestion)
          ↓
     Silver Layer
(Clean & Standardize)
          ↓
      Gold Layer
 (Star Schema Model)
          ↓
       Data Marts
```

# 🥉 Bronze Layer – Data Ingestion

## Objective

Load raw source files into PostgreSQL while preserving original data.

### Key Activities

* Built Python ingestion pipelines
* Loaded CSV and Excel sources
* Preserved source-level metadata
* Implemented batch tracking

### Metadata Columns

* _source_file
* _source_platform
* _ingested_at
* _batch_id

### Special Case: Sales Target Versioning

The sales target file contains multiple planning versions:

* Plan_v1_Original
* Plan_v2_Adjustment_H2

All target versions are preserved without overwriting historical records.

## Sales Target Versioning Logic

The project implements historical target planning logic.

| Period     | Effective Version |
| ---------- | ----------------- |
| Month 1–6  | v1                |
| Month 7–12 | v2                |

Additional versioning attributes:

* effective_from
* effective_to
* is_latest

This enables historical tracking of target adjustments throughout the year.
